远程驾驶系统开发规范
（C++ + Qt + 视频流 + 跨设备通信）

文档版本：v1.0
编制日期：2025年
适用范围：基于C++语言、Qt界面、实时视频流和跨设备远程通信的复杂远程驾驶系统（含驾驶舱与车辆端）
目标：高效开发出高可靠、低延迟、可维护、安全稳健的工业级远程驾驶系统

1. 引言
1.1 系统概述
远程驾驶系统是典型的安全关键分布式实时系统，包含以下核心特性：

高性能C++核心实现
Qt/QML作为人机交互界面
低延迟双向实时通信
高清低延迟视频流（H.265/WebRTC）
严格的实时性与安全冗余要求
1.2 核心挑战
端到端延迟控制（目标 < 150ms）
复杂状态管理与容错
网络环境不稳定下的可靠通信
安全关键场景下的功能安全与网络安全
团队协作下的代码可维护性
1.3 本规范目的
规范从架构、编码、通信、测试到工程化全流程提出强制性要求，确保系统兼具开发效率与工业级稳健性。

2. 总体架构规范
必须遵循以下原则：

严格分层架构（禁止跨层直接调用）：

表现层（Qt/QML）
应用服务层（状态机、业务编排）
领域核心层（决策、感知融合、控制算法）
基础设施层（通信、视频编解码、时间同步、日志）
平台抽象层（不同硬件、OS适配）
契约优先设计：所有模块间、设备间接口必须先定义Schema（推荐FlatBuffers）。

命令控制流与视频流物理隔离（不同Socket或不同QoS通道）。

事件驱动 + 分层状态机架构：全系统使用状态机管理（推荐Boost.SML或Qt SCXML）。

驾驶舱端与车辆端采用相同领域模型，减少语义偏差。

采用“影子模式”设计：所有关键决策必须同时在仿真环境与真实环境并行运行。

3. C++ 编程规范
3.1 语言与标准
必须使用 C++20（推荐 C++23）
必须关闭异常（-fno-exceptions）
必须遵循 MISRA C++ 2023 + AUTOSAR C++14 核心规则
强烈推荐参考《C++ Core Guidelines》
3.2 强制要求
严禁使用裸指针（new/delete），必须使用智能指针或对象池
核心控制路径禁止使用 std::shared_ptr（性能敏感）
必须使用内存池管理视频帧、网络包等高频对象
所有未定义行为（UB）必须通过静态分析消除
必须启用高警告级别编译：-Wall -Wextra -Werror -Wpedantic
3.3 代码质量工具链
静态分析：clang-tidy + MISRA检查器 + CppCheck
动态分析：AddressSanitizer、ThreadSanitizer、Valgrind
代码格式：统一使用 .clang-format
4. Qt 开发规范
优先使用 QML + Qt Quick，Widgets仅用于工具类界面
严格遵守 MVVM 模式：业务逻辑全部下沉到C++服务层
所有耗时操作必须移至工作线程（QObject::moveToThread）
信号槽跨线程时必须显式使用 Qt::QueuedConnection
使用 QStateMachine 或 Boost.SML 管理复杂界面状态
必须实现界面与核心逻辑的完全解耦，便于自动化测试
视频显示推荐使用 QQuickFramebufferObject 或自定义 QVideoSink
5. 通信与协议规范
5.1 技术选型（推荐）
控制指令：FlatBuffers over QUIC 或 WebRTC DataChannel
视频流：WebRTC（首选）或 SRT + H.265
序列化：FlatBuffers（必须支持Schema演进）
5.2 必须实现的功能
双通信链路冗余（主备切换 < 500ms）
心跳 + 超时安全停车机制（默认超时5秒）
指令序列号、时间戳、防重放保护
带宽自适应与智能丢帧策略
所有通信必须加密（DTLS/TLS 1.3）
断线后自动进入安全降级模式
6. 视频流处理规范
必须使用硬件编码器（NVENC、AMF、Jetson NVENC、RKMPP等）
编码器参数必须极致低延迟调优（zerolatency、slices、vbv-maxrate）
推荐使用 GStreamer 或 FFmpeg 构建管道
必须实现自定义扩展头携带帧类型、丢帧提示、PTS等信息
必须支持动态分辨率与码率切换
端到端延迟必须实时监控并在界面显示
7. 并发、实时性与性能规范
必须设计清晰的线程模型：

控制线程（最高优先级）
视频编解码线程
网络IO线程（推荐io_uring）
Qt GUI线程（仅负责渲染）
严禁在控制路径上使用互斥锁，优先使用 lock-free 队列（SPSC Ring Buffer）

关键路径避免内存分配（全部使用内存池）

必须实现高精度时间同步（PTP或自定义时间服务）

必须对所有关键路径进行延迟基准测试并记录基线

8. 可靠性、安全与功能安全规范
8.1 可靠性
必须实现 Fail-Safe 与 Fail-Operational 机制
必须有完整的黑匣子记录系统（输入、输出、视频、状态）
必须进行定期故障注入测试（网络丢包、延时突变、编码器崩溃等）
8.2 安全
必须完成 STRIDE 威胁建模
参考 ISO 26262（功能安全）和 ISO/SAE 21434（网络安全）开发流程
所有外部输入必须进行严格校验
采用零信任架构思想进行设备认证
9. 测试与验证规范
必须建立以下五层测试体系：

单元测试（GoogleTest + GMock，核心模块覆盖率 ≥ 90%）
契约测试（接口Schema验证）
集成测试（模块间通信）
硬件在环（HIL）仿真测试（最重要）
真实车辆闭环测试（含影子模式）
强制要求：

核心控制代码分支覆盖率 ≥ 85%
必须建立自动化回归测试流水线
必须有独立的仿真测试环境（支持录制回放）
10. 工程化与 DevOps 规范
构建系统：CMake 3.22+（现代语法）
依赖管理：Conan 2（推荐）或 vcpkg
版本控制：Trunk Based Development + Feature Flag
代码审查：所有代码必须经过至少两人Review
CI/CD：必须包含静态分析、单元测试、代码覆盖率检查
文档要求：使用 ADR（Architecture Decision Records） 记录重要决策
日志：使用 spdlog + 结构化JSON日志
11. 推荐技术栈（2025版）
核心：C++23 + CMake + Conan2
UI：Qt 6.7+（QML为主）
通信：WebRTC (libwebrtc) + QUIC + FlatBuffers
日志：spdlog
状态机：Boost.SML
序列化：FlatBuffers
视频：FFmpeg / GStreamer + 硬件编码器
测试：GoogleTest + GMock + Catch2
12. 强制检查清单（发布前必须全部通过）
 架构分层是否被破坏？
 是否全部通过 MISRA C++ 检查？
 控制链路是否与视频流物理隔离？
 是否存在不必要的 std::shared_ptr 或互斥锁？
 关键路径是否全部使用内存池？
 是否实现双链路冗余与超时安全停车？
 端到端延迟是否实时监控并显示？
 是否完成威胁建模并有应对措施？
 核心模块单元测试覆盖率是否达标？
 是否有完整的黑匣子记录系统？
