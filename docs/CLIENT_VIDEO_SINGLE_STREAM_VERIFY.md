# 单路视频流花屏验证指南

## 目标

先在一路视频流下验证无花屏，再验证四路。避免四路并发干扰排查。

## 单路验证步骤

### 1. 只连接一路视频

- 在客户端选择**仅带单路摄像头**的车辆/会话，或
- 在车端/媒体服务侧临时只推一路 H.264（例如只推前视），使客户端只建立一条 WebRTC 视频轨道。

这样只会创建一个 `H264Decoder`，无多路 concealing 计数交叉。

### 2. 运行并观察

```bash
# 编译
make build-client

# 运行（如需看 H264 日志可重定向）
./RemoteDrivingClient 2>&1 | tee /tmp/h264.log
```

观察：

- 若出现 **「用上一帧替代」**、**「丢弃有损帧(无 lastGood)」**：说明有损帧被正确替换或丢弃，未送显。
- 若 **「统计 出帧= … 替代= …」** 里 **替代 > 0**：说明花屏帧已被上一帧替代。
- 若仍花屏：查看日志里 **「帧有损」**、**「concealing_delta」**、**「RTP 重排缓冲满」**、**「丢弃 P(等IDR)」** 的频率，判断是丢包/乱序导致频繁 flush，还是解码路径仍有个别坏帧未替换。

### 3. 当前防花屏逻辑（单路相关）

| 逻辑 | 说明 |
|------|------|
| 有损且存在上一帧 | 用上一帧替代输出，不显示本帧 |
| 有损且无上一帧 | 丢弃本帧不输出，避免首帧或 flush 后首帧花屏 |
| RTP 重排缓冲 | 128 包，减少乱序导致的 flush，单路更稳 |
| 丢包/缺口 | 仍会 flush 并丢弃 P 直至下一 IDR，属预期 |

### 4. 单路通过后再测四路

单路无花屏后，再连接四路摄像头，确认：

- 四路均有「用上一帧替代」或「丢弃有损帧」时画面正常；
- 若某路仍花屏，对比该路日志的 **concealing_delta**、**替代** 与 **出帧** 比例。

## 常见现象

- **大量「丢弃 P(等IDR)」**：网络有丢包或乱序，频繁触发 flush；可查网络或适当加大服务端 IDR 间隔（需权衡延迟）。
- **替代=0 且无「用上一帧替代」**：concealing 未与当前 decoder 匹配或全局计数未增加；已用全局 fallback，若仍为 0 需抓日志看 `had_concealing` 与 `global_after > global_before`。
- **首帧或短暂花屏**：首帧有损且无 lastGood 时已改为丢弃不输出，若仍出现请确认运行的是最新构建。
